# Social Commerce Harness

这套 harness 面向“社交媒体账号 + 电商平台店铺”的运营场景，假设业务主要在平台内成交，没有独立网站。

本目录下的文件不再只是角色说明书，而是“可直接给 Agent 使用”的 system prompt 风格文档。它们仍然保持人类可读，但写法更偏向可执行指令。

这套角色同时采用一套展示层 IP 设定，统一归属于“林地增长联盟”。

- 这套命名用于增强角色辨识度和协作感
- 它是展示层别名，不替代原有文件名、正式角色名和职责边界

## 使用方式

- 如果你想让某个 Agent 扮演具体角色，直接把对应文件内容作为系统提示词或角色提示词使用。
- 如果你是人工主导，可以把这些文件当作角色卡和协作规则来阅读。
- 如果你要做多 Agent 编排，建议把 `orchestrator` 作为默认入口调度器，再把 `账号主理 / 内容策略官` 作为业务策略主控角色按需调用。

## Skill 接入说明

- 当前这套 harness 已接入第一梯队和第二梯队 skill，可直接作为各角色工作时的参考规则。
- 各角色具体引用哪些 skill、哪些是核心 owner、哪些只是协作引用，请统一查看 `skill-assignment.md`。
- 如果你想快速开始，先从 `orchestrator` 进入；如果你想看完整分配口径，再打开 `skill-assignment.md`。
- 当前 `agents/*.md` 里的 `Skill 使用规则` 已与 `skill-assignment.md` 对齐，后续新增 skill 也建议先更新总表，再回填到各角色。

## 角色清单

- `01-account-strategist.md`：账号主理 / 内容策略官（白狐策略师）
- `02-content-producer.md`：内容制作官（喜鹊制作人）
- `03-store-operator.md`：店铺运营官（三花猫店掌柜）
- `04-assortment-supply-manager.md`：选品与供给官（松鼠管仓）
- `05-growth-amplification-manager.md`：投放与增长官（兔子增长手）
- `06-insights-cx-analyst.md`：数据复盘与客服体验官（猫鼬观察员）

## 推荐主从关系

- 默认入口调度：`orchestrator`（猫头鹰召集人）
- 业务主控角色：`账号主理 / 内容策略官`
- 支撑角色：`选品与供给官`、`店铺运营官`
- 执行角色：`内容制作官`、`投放与增长官`
- 反馈角色：`数据复盘与客服体验官`

## 推荐协作顺序

以下是典型全链路业务流程，不代表所有任务的默认入口；实际入口、停机规则和路由顺序以 `orchestrator` 为准。

```text
orchestrator 判断任务类型与信息是否充分
-> 账号主理定目标
-> 选品与供给确认可推商品
-> 店铺运营准备承接
-> 内容制作批量产出素材
-> 投放与增长放大有效内容/商品
-> 数据复盘与客服体验回流问题和机会
-> 账号主理更新下一轮策略
```

## 推荐输入上下文

给这些 Agent 的上下文最好覆盖 5 类信息：

- 内容数据：曝光、完播、互动、进店、点击、成交
- 店铺数据：访客、加购、下单、支付、退款、客单、毛利
- 商品数据：库存、发货时效、缺货、售后风险、价格带
- 用户反馈：评论、私信、客服对话、差评、退款原因
- 平台环境：活动节点、平台规则、竞品动态、热点内容

## 统一约束

所有角色都应遵守以下原则：

- 不要编造库存、利润、投放效果、用户反馈等关键事实
- 不要越过自己的角色边界代替别的角色做最终判断
- 当输入不足时，先明确缺什么，再在现有信息范围内给暂定建议
- 输出尽量结构化，便于交接给其他角色或复盘
- 所有建议都应服务于“内容 -> 承接 -> 成交 -> 复盘”的闭环
