# CLAUDE Orchestrator Routing Design

**日期**：2026-04-22  
**范围**：把仓库级 `CLAUDE.md` 从“skill 仓库说明”改造成“社交媒体电商多 Agent 环境的总控文档”，让用户与 Claude 的自然语言业务对话默认先进入 `orchestrator`，再由其将任务交给具体业务角色承接。  
**目标**：让 `CLAUDE.md` 成为真实可执行的协作入口，而不是仅描述仓库结构；用户无需记忆复杂命令，就能通过自然语言让合适的 agent 接手工作，并清楚看到当前路由状态。

## 背景

当前仓库已经具备以下基础：

- `agents/orchestrator.md`：定义了任务识别、停机规则、默认路由顺序和输出格式
- `agents/01-account-strategist.md` 到 `agents/06-insights-cx-analyst.md`：定义了各角色的职责、输入、输出、边界和 skill 使用规则
- `agents/skill-assignment.md`：定义了 skill owner 与协作引用边界

但当前 `CLAUDE.md` 仍然基本是原始 skill 仓库说明，核心内容集中在：

- 仓库结构
- skill frontmatter 规范
- marketplace 说明
- Claude Code 的 `!` 命令注入技巧

这导致一个问题：仓库内部已经形成多 agent 协作体系，但用户和 Claude 的对话入口并没有切换到这套体系。结果是：

1. 用户直接下达业务任务时，Claude 不一定先走 `orchestrator`
2. Claude 容易自己直接回答，而不是把任务交给对应业务角色
3. 用户无法明确感知当前是谁在承接任务、为什么这样路由、缺什么信息

## 本轮设计目标

本轮改造只处理 `CLAUDE.md` 的入口协议，不改动 `agents/*.md` 和 `skills/*` 的角色与能力定义。

需要满足 5 个目标：

1. 用户使用自然语言下达业务任务时，默认先进入 `orchestrator`
2. `orchestrator` 必须显式展示任务分类、信息充分性、缺失信息和建议承接角色
3. 具体结论和产出必须由对应业务角色承接，避免 Claude 自己吞掉所有工作
4. 用户可以通过显式点名某个角色来覆盖默认路由
5. 普通闲聊、仓库维护问题、纯文档编辑类任务不应强制进入业务 agent 流程

## 用户已确认的交互原则

本次对齐后，入口协议遵循以下用户确认：

- 默认先经 `orchestrator`
- 路由过程对用户显式展示
- 以自然语言作为默认触发方式
- 用户明确点名某个 agent 时，可以覆盖默认路由

## 方案选择

本轮采用“总控协议化”方案，而不是“轻规则补充”。

### 不采用的方案 1：轻规则接入

只在 `CLAUDE.md` 补充少量说明，例如：

- 遇到业务任务先参考 `agents/orchestrator.md`
- 根据任务类型参考对应 agent 文档

问题：

- 约束力不足，Claude 仍可能直接给综合答案
- 无法形成稳定的用户可见交接流程
- 容易与现有 `agents/orchestrator.md` 的细致规则脱节

### 不采用的方案 2：角色直连优先

按关键词把用户任务直接路由到某个具体业务角色，只在复杂任务时才补走 `orchestrator`。

问题：

- 会削弱 `orchestrator` 作为统一入口的价值
- 在跨角色任务中容易过早进入单一角色，导致遗漏停机条件和上游依赖
- 与用户希望的“所有业务任务先调度再承接”不一致

### 采用的方案：总控协议化

`CLAUDE.md` 明确规定：

- 哪些输入属于业务任务
- 业务任务一律先经过 `orchestrator`
- `orchestrator` 只负责识别、停机、分发和汇总
- 具体分析与产出必须交给对应角色
- 对用户展示当前承接 agent、路由原因和下一步

## CLAUDE.md 的新职责

改造后，`CLAUDE.md` 的职责应从“仓库说明”切换为“协作入口协议”。

它主要负责：

1. 定义当前项目是一个社交媒体电商多 agent 工作环境
2. 定义用户与 Claude 的默认协作模式
3. 定义哪些任务需要先进入 `orchestrator`
4. 定义如何显式展示承接角色与路由状态
5. 定义何时允许用户绕过默认路由
6. 把具体角色职责与 skill 规则委托给 `agents/*.md`

它不负责：

- 重新描述每个 agent 的全部职责细节
- 重新维护 skill owner 与协作边界
- 复制 `orchestrator.md` 的完整任务分类细节

## 新的交互模型

### 输入类型划分

`CLAUDE.md` 需要把用户输入分成 3 类：

#### 1. 业务任务

典型特征：

- 讨论内容、店铺、选品、投放、复盘、活动、增长、客服体验
- 目标是拿到运营判断、执行方案、脚本、页面优化、复盘结论等业务输出

默认处理：

- 先进入 `orchestrator`

#### 2. 显式角色任务

典型特征：

- 用户明确说“让内容制作官来做”
- 用户明确说“直接交给店铺运营官”
- 用户明确说“不要调度，直接让账号主理判断”

默认处理：

- 允许跳过默认路由，直接按用户指定角色承接
- 但如果缺少该角色完成任务的必要信息，仍需按该角色规则停机补信息

#### 3. 非业务任务

典型特征：

- 维护仓库文件
- 修改 prompt 文档
- 调整 skill 文案
- 讨论项目结构、实现方式、脚本逻辑

默认处理：

- 不进入业务 agent 路由
- 直接按普通开发/文档任务处理

## Orchestrator 入口规则

### 默认入口条件

当输入满足以下任一条件时，先进入 `orchestrator`：

- 用户在问“这周怎么做”
- 用户在问“发什么内容”
- 用户在问“主推什么商品”
- 用户在问“店铺怎么承接”
- 用户在问“要不要投流 / 放量 / 合作”
- 用户在问“为什么成交掉了 / 退款多了 / 转化差了”

### 入口后的标准动作

`orchestrator` 进入后先做 4 件事：

1. 识别任务类型
2. 判断信息是否充分
3. 判断是否需要停机补信息
4. 确定建议承接角色与顺序

### 入口后的展示格式

`CLAUDE.md` 应要求 Claude 在对用户可见的回复中显式展示以下内容：

```text
当前承接：
- orchestrator

任务类型：
- 周计划 / 内容生产 / 店铺承接 / 放量增长 / 复盘诊断

信息是否充分：
- 充分 / 不充分

缺失信息：
- ...

建议交接：
- 下一位承接 agent
- 该 agent 需要解决的问题
```

这一步的目的不是把内部思考全部暴露，而是让用户明确知道系统当前处于哪个环节。

## 角色承接规则

`CLAUDE.md` 不重写角色 prompt，而是建立委托关系。

### 委托原则

- 周目标、主推方向、内容支柱类问题委托给 `01-account-strategist.md`
- 脚本、标题、图文结构、内容版本类问题委托给 `02-content-producer.md`
- 商品页、活动页、利益点、承接链路类问题委托给 `03-store-operator.md`
- SKU 优先级、供给、履约、货盘适配类问题委托给 `04-assortment-supply-manager.md`
- 投流、放量、预算、达人、实验类问题委托给 `05-growth-amplification-manager.md`
- 数据复盘、客服反馈、评论洞察、流失与退款问题委托给 `06-insights-cx-analyst.md`

### 边界原则

- Claude 不直接越过角色边界给完整专业结论
- Claude 不把多个角色职责压缩成一段“万能答案”
- Claude 可以做汇总，但汇总前必须明确承接来源和顺序

## 用户可见的交接协议

为了让用户感知“是谁在接活”，`CLAUDE.md` 需要要求以下行为：

### 显式说明当前角色

当完成路由后，输出中应明确当前由谁承接，例如：

- 当前由 `账号主理 / 内容策略官` 承接策略判断
- 当前由 `内容制作官` 承接内容生产
- 当前由 `店铺运营官` 承接转化承接优化

### 显式说明切换原因

需要简要说明为什么切到该角色，例如：

- 因为当前问题是“本周推什么”，先由 `账号主理` 负责目标与主推方向
- 因为当前问题是“这个商品页怎么接内容流量”，改由 `店铺运营官` 处理

### 显式说明下一步

多角色任务需要说明下一步，例如：

- 先由 `选品与供给官` 确认可推 SKU，再交给 `内容制作官` 出脚本
- 先由 `数据复盘与客服体验官` 明确问题，再交给 `账号主理` 调整下一轮策略

## 用户覆盖默认路由的规则

### 允许覆盖的情形

当用户明确说出以下意图时，允许跳过 `orchestrator`：

- “直接让内容制作官来做”
- “不要调度，店铺运营官直接判断”
- “这个问题只给投放与增长官看”

### 覆盖后的约束

即使跳过默认路由，也仍需遵守：

- 指定角色自己的输入要求
- 指定角色自己的边界
- 信息不足时先停机补信息

这意味着：

- 用户可以指定承接角色
- 但不能强迫角色在缺关键条件时编造结论

## 非业务任务的排除规则

以下任务不进入业务 agent 流程：

- 修改 `CLAUDE.md`
- 编辑 `agents/*.md`
- 调整 `skills/*/SKILL.CN.md`
- 讨论仓库目录结构
- 编写脚本、工具或验证命令

这些任务应按普通开发与文档协作处理，避免把 agent 路由系统错误地套到仓库维护工作上。

## 文档结构设计

改造后的 `CLAUDE.md` 建议采用以下结构：

1. 项目定位
2. 工作模式
3. 默认业务任务入口
4. `orchestrator` 的职责与限制
5. 角色交接规则
6. 用户覆盖默认路由的方法
7. 非业务任务的处理方式
8. 相关文档索引

## 风险与控制

### 风险 1：`CLAUDE.md` 和 `orchestrator.md` 重复维护

控制方式：

- `CLAUDE.md` 只定义入口协议
- 任务分类、停机规则和默认顺序继续以 `orchestrator.md` 为准

### 风险 2：`CLAUDE.md` 写得过长，重新变成说明书

控制方式：

- 删除不再适合当前项目定位的旧 skill marketplace 说明
- 避免把角色职责全文复制到 `CLAUDE.md`
- 用“入口协议 + 文档索引”替代“全量内容堆叠”

### 风险 3：业务任务和仓库维护任务混淆

控制方式：

- 单独增加“非业务任务排除规则”
- 明确只有社交媒体电商经营任务才进入 agent 路由

### 风险 4：用户感觉不到当前由谁承接

控制方式：

- 在 `CLAUDE.md` 中强制要求显式展示当前承接角色、路由原因和下一步

## 验收标准

完成后应满足：

1. `CLAUDE.md` 开头能够清楚说明当前项目是社交媒体电商多 agent 工作环境
2. 用户自然语言提出业务任务时，默认先进入 `orchestrator`
3. 对用户可见的回复中包含当前承接角色和路由信息
4. 用户明确点名角色时，可以覆盖默认路由
5. 仓库维护类任务不会误触发业务 agent 流程
6. `CLAUDE.md` 不重复复制 `agents/*.md` 的细节，而是以它们作为权威引用
