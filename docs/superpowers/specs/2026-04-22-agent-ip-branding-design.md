# Agent IP Branding Design

**日期**：2026-04-22  
**范围**：为当前社交媒体电商多 Agent 项目建立一套统一的动物 IP 设定，包括角色集合名称、各 agent 的动物形象与头衔风格，以及后续在入口文档和角色文档中的落地原则。  
**目标**：让项目从“功能角色集合”升级为“有辨识度的角色联盟”，在不破坏现有职责边界的前提下，增强记忆点、协作感和对外表达的一致性。

## 背景

当前仓库已经完成从原始 skill 汇集仓库向“社交媒体账号 + 电商平台店铺”的多 Agent 工作环境转型，具备以下特点：

- `CLAUDE.md` 负责统一入口协议
- `agents/orchestrator.md` 负责业务任务的路由、停机、交接和汇总
- `agents/01-account-strategist.md` 到 `agents/06-insights-cx-analyst.md` 负责具体业务角色

目前这套体系已经具备明确的职责划分，但对外呈现仍然偏“说明书式角色列表”，主要问题是：

1. 角色记忆点主要来自文件名和职能描述，缺少统一人格化识别
2. `orchestrator + 6 个业务角色` 之间有协作关系，但缺少一个能承载团队感的总称
3. 如果未来要在 README、入口文档、图标、插画或介绍页中展示这套体系，当前角色名称不够 IP 化

## 本轮设计目标

本轮只做 IP 设定层设计，不改动角色职责与协作边界。

需要满足 5 个目标：

1. 给整套 agent 取一个统一、可持续使用的集合名称
2. 为每个角色匹配一个人格感明确、职责贴合的动物形象
3. 让头衔比“官”更亲切，但仍然保留专业分工辨识度
4. 保持“林地”世界观一致，避免动物来源和整体设定打架
5. 后续可平滑落地到 `CLAUDE.md`、`agents/README.md` 和各角色文档中

## 用户已确认的设计偏好

本轮设计基于以下确认逐步收敛：

- 整体气质采用“专业智囊型”
- 角色命名采用“动物人格 + 明确岗位”的方式
- 集合名称更偏“组织 / 联盟名”，而不是单个产品名或纯世界观名
- 头衔不统一使用“主理人”，避免全员同模板造成层次单一
- 角色动物尽量保持林地体系内，并优先选择地面动物或与林地不冲突的角色
- 增长角色避免过强攻击性

## 方案选择

本轮采用“林地联盟 + 混合头衔”的方案，而不是“统一官职”或“纯萌系吉祥物”。

### 不采用的方案 1：统一官职体系

示例方向：

- 猫头鹰调度官
- 白狐策略官
- 河狸供给官

问题：

- 太像传统组织架构，亲和度不足
- 所有角色语气过于整齐，缺少团队内部分工的层次
- 不利于形成“创业小队”而非“正式机构”的气质

### 不采用的方案 2：统一主理人体系

示例方向：

- 白狐策略主理人
- 蜂鸟内容主理人
- 河狸供给主理人

问题：

- 虽然比“官”亲切，但全员同后缀会显得发闷
- 各角色之间的岗位差异感被削弱
- 对 `orchestrator`、店铺、增长、复盘等角色不够贴脸

### 不采用的方案 3：纯可爱动物吉祥物体系

示例方向：

- 所有角色都强调“萌感”和可爱度

问题：

- 容易削弱“策略、履约、增长、复盘”这些岗位的专业感
- 和当前项目作为工作环境、协作框架的定位不完全匹配

### 采用的方案：林地增长联盟

采用“联盟名 + 动物人格 + 混合职能头衔”的方案：

- 用 `林地增长联盟` 作为总集合名
- 用动物承载人格与行为意象
- 用不同头衔体现不同岗位分工
- 保持整体亲切、协作、专业，而非攻击性或官僚感

## 集合名称设计

### 最终名称

- `林地增长联盟`

### 选择理由

- `林地` 提供统一世界观，方便后续继续补充视觉和故事设定
- `增长` 直接保留项目的业务目标导向，不会只变成空泛的动物世界
- `联盟` 自带协作关系，适合 `orchestrator + 多角色` 的结构

### 不采用的备选方向

- `林地智囊团`：更偏策略咨询感，适合前期方案，但弱于“增长”这个业务目标
- `森域策士团`：有风格，但过于书面，落地到文档会稍显远
- `林间作战局`：动作感强，但“作战”偏硬，不够亲和

## 命名规则

### 核心公式

每个角色采用：

- `动物 + 头衔`

其中：

- 动物负责传达人格、行为联想和记忆点
- 头衔负责传达职能边界和岗位感

### 头衔规则

不要求统一后缀，而采用“偏创业团队”的混合称谓：

- `召集人`
- `策略师`
- `制作人`
- `店掌柜`
- `管仓`
- `增长手`
- `观察员`

这样可以同时满足：

- 比“官”更平易近人
- 比“搭子”更专业
- 比“全员主理人”更有岗位差异

## 最终角色设定

### 1. `orchestrator`

- 名称：`猫头鹰召集人`
- 对应职责：任务识别、信息充分性判断、停机、路由、交接、汇总
- 选择理由：猫头鹰具备全局观察与夜间巡察的意象，适合只做轻量调度、不直接吞掉全部业务工作

### 2. `01-account-strategist`

- 名称：`白狐策略师`
- 对应职责：账号定位、周目标、主推方向、内容支柱
- 选择理由：白狐灵活、聪明、擅长判断形势，适合承担策略规划与优先级取舍

### 3. `02-content-producer`

- 名称：`喜鹊制作人`
- 对应职责：脚本、标题、图文结构、内容版本
- 选择理由：喜鹊天然带有表达、传递、包装与传播联想，适合内容产出角色

### 4. `03-store-operator`

- 名称：`刺猬店掌柜`
- 对应职责：商品页、活动页、利益点、承接链路、可用性检查
- 选择理由：刺猬给人的感觉是细致、谨慎、守住边界，适合承担看店、查漏补缺和承接体验优化

### 5. `04-assortment-supply-manager`

- 名称：`松鼠管仓`
- 对应职责：SKU 优先级、备货、库存、履约、货盘匹配
- 选择理由：松鼠天然对应囤货、分配、储备和周转，和供给、仓储、节奏管理高度贴合

### 6. `05-growth-amplification-manager`

- 名称：`兔子增长手`
- 对应职责：投流、放量、预算、达人协同、增长实验
- 选择理由：兔子敏捷、轻快、地面系，适合强调快速试错、迅速响应与放量动作，同时避免猛禽类的攻击性

### 7. `06-insights-cx-analyst`

- 名称：`猫鼬观察员`
- 对应职责：评论洞察、客服体验、退款、流失、复盘诊断
- 选择理由：猫鼬警觉、敏锐、善于发现异常信号，适合承担反馈回流和问题识别

## 角色组合的整体气质

最终组合应呈现以下统一印象：

- 是一支林地里的协作小队，而不是冰冷的组织架构图
- 角色彼此有分工，但不会给人“官僚系统”感
- 可用于日常文档命名，也可继续发展为头像、插画、介绍页等视觉资产
- 保持业务专业度，不会因为动物设定而削弱工作可信度

## 在文档中的落地方式

### `CLAUDE.md`

建议作为入口协议继续保留现有角色正式名，同时在合适位置补充轻量映射，例如：

- `orchestrator`（猫头鹰召集人）
- `账号主理 / 内容策略官`（白狐策略师）

原则：

- `CLAUDE.md` 仍以入口协议为主
- 不把 IP 设定写成大段故事
- 只在首次介绍、角色索引或补充说明中出现昵称映射

### `agents/README.md`

建议在角色清单中增加一列或补充括号说明，让用户同时看到正式职能名与 IP 名：

- `01-account-strategist.md`：账号主理 / 内容策略官（白狐策略师）
- `03-store-operator.md`：店铺运营官（刺猬店掌柜）

这是最适合集中展示整套 IP 的位置。

### `agents/*.md`

建议采用“轻映射，不改职责”的方式：

- 保留原文件名和正式角色名
- 在文件开头的简介或别名处补充 IP 称呼
- 不改动各文档对输入、输出、边界和停机规则的定义

### 后续扩展

如果未来继续 IP 化，可以在不改变当前命名体系的前提下补充：

- 统一头像风格
- 每个角色的代表色
- 一句角色口头禅
- README 中的联盟介绍区块

## 风险与控制

### 风险 1：IP 名压过正式角色名

控制方式：

- 文件名、职责边界、交接规则继续沿用正式角色名
- IP 名作为别名或展示层补充，不替代系统结构

### 风险 2：动物世界观不统一

控制方式：

- 优先保持林地语义一致
- 避免出现海洋、极地、猛禽等与整体气质冲突的动物

### 风险 3：头衔体系再次变得过于整齐

控制方式：

- 保持混合称谓，不统一替换为同一后缀
- 每个头衔都由岗位语义驱动，而非仅追求形式统一

### 风险 4：角色可爱化后削弱专业度

控制方式：

- 所有动物选择都必须先满足职责贴合，再考虑可爱或记忆点
- 避免使用过于玩具化、幼态化的角色名称

## 验收标准

完成后应满足：

1. 项目拥有统一的角色集合名称 `林地增长联盟`
2. `orchestrator + 6 个业务角色` 均有明确、稳定的动物 IP 映射
3. 头衔语气亲切但不失专业，不统一使用“官”或“主理人”
4. 各动物与岗位职责具备清晰、可解释的对应关系
5. 设定能够无缝落到 `CLAUDE.md`、`agents/README.md` 和各角色文档中，而不破坏现有结构
